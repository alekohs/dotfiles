Vibe-code this, improving it in tight self-correcting cycles: $@

You're a local model — drift is the enemy, so work mechanically:

1. **Anchor on disk.** Keep vision, run command, "works now", and a prioritized backlog in `.vibe/PLAN.md`. Create it if missing (scaffold the smallest thing that _runs_); re-read it at the start of every cycle. The file is your memory, not this chat.
2. **One thing per cycle.** Take the single top backlog item, make the smallest change that completes it, touching as few files as possible.
3. **Verify by running**, not by re-reading the diff. If it breaks and you can't fix it in 2 tries, revert to the last commit and try a smaller step. Never end a cycle broken.
4. **Critique, then enqueue.** Ask: does the core flow work end-to-end? is the main action obvious? worst rough edge? riskiest untested path? Turn the sharpest answers into _concrete_ backlog items — never "improve UX".
5. **Snapshot** with a git commit each working cycle, update the plan, then repeat.

Run ~5 cycles or until no `[must]` items remain, then report what works and what's left. For the full version with tool scoping use the `vibe` skill.
